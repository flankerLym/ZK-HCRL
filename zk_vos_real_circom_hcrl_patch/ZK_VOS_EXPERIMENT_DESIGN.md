# ZK-VOS Experiment Design

## Experimental goal

The existing HCRL experiments evaluate scheduling quality. ZK-VOS evaluates **verifiability and deployability**:

> Can an off-chain HCRL scheduling result be verified on-chain without revealing private audit scores and full oracle-state details?

## Experiment 1: Compliance correctness

Construct valid and invalid schedule records.

| Case | Modification | Expected result |
|---|---|---|
| Valid | all constraints satisfied | accepted |
| Low reputation | `repEff < reputationThreshold` | rejected |
| Over cost | `cost > costBudget` | rejected |
| Over risk | `risk > riskBudget` | rejected |
| Over latency | `latencyEst > deadline` | rejected |
| Cooldown | `cooldown = 1` | rejected |
| Service mismatch | oracle service type does not match request type | rejected |
| Invalid membership | wrong Merkle path/root | rejected |

Metrics:

- Valid Acceptance Rate
- Invalid Rejection Rate
- False Accept Rate
- False Reject Rate

## Experiment 2: ZK overhead

Metrics:

- Circuit constraints
- Witness generation time
- Proof generation time
- Proof size
- Off-chain verification time
- Solidity verification gas
- Submit schedule gas
- Verifier deployment gas

## Experiment 3: Constraint-combination ablation

Use four conceptual versions in the paper:

| Variant | Constraints |
|---|---|
| ZK-Basic | reputation only |
| ZK-Budget | reputation + cost + risk |
| ZK-SLA | reputation + cost + risk + latency + cooldown |
| ZK-Full | ZK-SLA + service match + Merkle membership |

The provided `zk_vos_full.circom` implements the full version. To create smaller variants, remove constraints from the circuit or pass neutral budgets.

## Experiment 4: Oracle pool scalability

Vary oracle pool size:

- 30
- 60
- 120
- 240

Expected interpretation:

- On-chain verification does not scan the entire oracle pool.
- Membership is represented by a Merkle root and path.
- Verification overhead grows slowly with Merkle depth rather than linearly with pool size.

## Experiment 5: Prototype deployment

Deploy:

- `Verifier.sol` generated by snarkjs
- `OracleScheduleRegistry.sol`

Then submit:

- valid proof
- invalid proof or invalid public signals

Report accepted/rejected events and gas.
